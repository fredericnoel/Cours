# Accessibilité - Inclusion et prise en charge des développeurs en situation de handicap

## Objectifs pédagogiques
À l’issue du cours, l’étudiant doit être capable de :
- comprendre ce que recouvre le handicap dans le contexte du développement web ;
- connaître le cadre légal français ;
- identifier les freins techniques, organisationnels et humains ;
- adopter une posture professionnelle inclusive ;
- proposer des aménagements réalistes dans une équipe technique ;
- éviter les erreurs courantes (invisibilisation, sur-adaptation, stigmatisation).

## Comprendre le handicap dans les métiers du développement
### Définition : handicap et travail dans le numérique
Définition légale (France), selon la loi du 11 février 2005 :
> « Constitue un handicap toute limitation d’activité ou restriction de participation à la vie en société subie dans son environnement par une personne… »

Point clé pour les développeurs : le handicap n’est pas uniquement visible, ni permanent.

### Typologies de handicaps rencontrés dans les équipes de développement
#### Handicaps moteurs
Exemples :
- troubles de la motricité fine ;
- usage limité ou impossible de la souris ;
- fatigue musculaire chronique.

Impacts possibles :
- difficulté avec certains claviers ;
- douleurs liées à de longues sessions ;
- besoin d’outils spécifiques (clavier ergonomique, commandes vocales).

#### Handicaps visuels
Exemples :
- malvoyance ;
- cécité partielle ;
- daltonisme.

Impacts concrets :
- usage de lecteurs d’écran (NVDA, VoiceOver) ;
- difficultés avec certains IDE / thèmes ;
- fatigue visuelle accrue.

#### Handicaps auditifs
Exemples :
- surdité partielle ;
- surdité totale ;
- presbyacousie.

Impacts :
- réunions orales non accessibles ;
- difficultés avec la communication informelle (_stand-up_, _daily_) ;
- dépendance aux supports écrits.

#### Handicaps cognitifs et neurodiversité
Exemples :
- dyslexie ;
- TDAH ;
- troubles de la concentration ;
- troubles anxieux.

Impacts possibles :
- difficulté avec la documentation dense ;
- surcharge cognitive ;
- gestion du stress, interruptions fréquentes.

Point important : ces profils sont présents dans le numérique, souvent non déclarés.

### Le handicap invisible : un enjeu majeur
- Environ 80 % des handicaps sont invisibles ;
- Beaucoup de développeurs ne déclarent pas leur situation ;
- Peur de
    - stigmatisation ;
    - frein à l'évolution ;
    - jugement sur les compétences.

Conséquence directe : l'environnement doit être inclusif par défaut, sans attendre une déclaration.

## Cadre légal et responsabilités professionnelles
### Obligations légales en France
- obligation d’emploi de travailleurs handicapés (OETH) ;
- aménagement raisonnable du poste de travail ;
- interdiction de discrimination (Code du travail).

Références :
- Loi 2005-102 ;
- Code du travail (articles L1132-1 et suivants).

### Rôle des acteurs institutionnels
- MDPH : reconnaissance du handicap (RQTH) ;
- Agefiph : aides financières, accompagnement ;
- Médecine du travail : préconisations d’aménagement.

### Ce que la loi n’impose pas
- pas d’obligation de révéler son handicap ;
- pas de justification détaillée auprès de l’équipe ;
- pas d’obligation de « performance supérieure ».

## Aménagements concrets dans un contexte de développement
### Aménagements techniques (sans baisse d’exigence)
#### Environnement de travail
- IDE personnalisable (thèmes contrastés, tailles de police) ;
- raccourcis clavier configurables ;
- outils de dictée ou de lecture ;
- écrans adaptés, claviers ergonomiques.

#### Organisation du travail
- flexibilité horaire (fatigue, soins) ;
- télétravail partiel ou total ;
- découpage clair des tâches ;
- priorisation explicite.

#### Documentation et communication
- documentation écrite claire ;
- décisions techniques tracées ;
- comptes rendus de réunion ;
- limitation de la communication implicite.

### Méthodes agiles et handicap
Bonnes pratiques reconnues :
- _user stories_ claires ;
- _definition of done_ explicite ;
- rituels adaptés ;
- outils collaboratifs accessibles.

## Posture professionnelle et erreurs à éviter
### Bonnes postures à adopter
- écouter sans supposer ;
- demander les besoins, pas les diagnostics ;
- proposer, ne pas imposer ;
- normaliser les aménagements.

### Erreurs fréquentes
- sur-protéger (infantilisation) ;
- sous-estimer les compétences ;
- forcer la déclaration du handicap ;
- considérer l’aménagement comme un « privilège » ;
- faire du handicap un sujet public sans consentement.

### Cas pratiques (discussions)
#### Cas 1 ####
Un développeur demande à éviter les réunions orales longues.
#### Cas 2 ####
Une développeuse refuse d’expliquer son aménagement à l’équipe.
#### Cas 3 ####
Un lead impose un outil non accessible à tous.
Analyse collective :
- que dit le droit ? ;
- que dit l’éthique ? ;
- quelle solution réaliste ?
